# Authentication Setup

wendRAG supports **optional Bearer token authentication** on the Streamable HTTP
transport (`/mcp`). Authentication is opt-in: when no keys are configured, the
server accepts unauthenticated requests (useful for local development). As soon
as a single key is registered, every request to `/mcp` must carry an
`Authorization: Bearer <token>` header.

The stdio transport is never gated by API keys -- trust is inherited from the
parent process that spawned `wend-rag stdio`.

## Threat model

- Keys are 32 bytes of cryptographically secure randomness from the operating
  system CSPRNG, hex-encoded, and prefixed with `wrag_` so leaked keys can be
  detected by secret-scanning tools.
- Only SHA-256 hashes are stored on disk; raw key material never touches the
  filesystem.
- The raw key is displayed to the operator **exactly once** at generation time.
- Key validation is constant-time to prevent timing side channels.
- On Unix, the keys file is written with `0o600` permissions (owner read/write
  only).

## Quick start

```bash
# 1. Generate a key (interactive)
wend-rag key:generate

# 2. Start the daemon
wend-rag daemon

# 3. Connect a client with Authorization: Bearer <token>
curl -H "Authorization: Bearer wrag_..." http://localhost:3000/mcp
```

## Generating a key

### Interactive

```text
$ wend-rag key:generate
Enter key name: RaymonKey
Generating...

Key Created

Name:       RaymonKey
Key:        wrag_8f3a2c1e9b4d7a6c5f8e1d3b9a2c4e7f1d8b3a6c9e2f5d8b1a4c7e0f3d6b9a2c
Prefix:     wrag_8f3a2c1e
Created at: 2026-04-17T18:42:15+00:00
Stored in:  /home/user/.wend-rag/keys.json

Keep this key safe -- it will not be shown again.
Use it with MCP clients by setting `Authorization: Bearer wrag_8f3a2c1e` on /mcp.
```

### Non-interactive (scripts)

```bash
wend-rag key:generate --name ci-runner
```

The command fails with a non-zero exit code if the name is empty or already
exists. Names must be unique so you can unambiguously revoke a specific key.

## Listing keys

```text
$ wend-rag key:list
NAME                     PREFIX               CREATED
RaymonKey                wrag_8f3a2c1e        2026-04-17T18:42:15+00:00
ci-runner                wrag_4c7e0f3d        2026-04-18T09:15:02+00:00
```

The `list` command only shows non-sensitive metadata (name, display prefix,
creation time). The underlying keys and hashes are never printed.

## Revoking a key

```bash
wend-rag key:revoke           # interactive: prompts for name
wend-rag key:revoke RaymonKey # non-interactive
```

Once revoked, any client still presenting the old key will receive
`401 Unauthorized`. Revocation takes effect the next time the daemon starts;
restart the server to pick up changes to `keys.json`.

## Authentication modes

wendRAG supports **three** modes, resolved at daemon startup:

### 1. No authentication (default)

If `WEND_RAG_API_KEY` is unset **and** `keys.json` has no entries (or is
missing), the server logs a warning and accepts all requests on `/mcp`.

```text
WARN API key authentication DISABLED -- /mcp accepts unauthenticated requests.
     Run `wend-rag key:generate` or set WEND_RAG_API_KEY to enable.
```

This is convenient for local development. **Do not** use this mode on any
reachable network.

### 2. Static key via environment variable

Set `WEND_RAG_API_KEY` to a single bearer token. Useful for container
orchestrators where injecting a secret env var is easier than mounting a file.

```bash
export WEND_RAG_API_KEY=wrag_your_static_token_here
wend-rag daemon
```

The env var can coexist with `keys.json`; both sets of hashes are checked.

### 3. Keys file with CLI management

Use `wend-rag key:generate`, `key:list`, and `key:revoke` to manage multiple
named keys in `keys.json`. This is the recommended mode for production: each
client gets a distinct, revocable key.

## Keys file location

The keys file lives at a platform-appropriate default:

| Platform | Default path |
|----------|-------------|
| Linux    | `$HOME/.wend-rag/keys.json` |
| macOS    | `$HOME/.wend-rag/keys.json` |
| Windows  | `%APPDATA%\wend-rag\keys.json` |

Override the path by setting `WEND_RAG_KEYS_FILE` to an absolute path:

```bash
export WEND_RAG_KEYS_FILE=/etc/wend-rag/keys.json
```

All three CLI subcommands and the daemon honour this override.

## Keys file schema

The file is a JSON array of `StoredKey` entries:

```json
[
  {
    "name": "RaymonKey",
    "key_prefix": "wrag_8f3a2c1e",
    "key_hash": "a1b2c3d4e5f6...64 hex chars total",
    "created_at": "2026-04-17T18:42:15Z"
  }
]
```

Do not edit this file by hand -- use the CLI. A handwritten hash that doesn't
match the canonical `SHA-256(wrag_<body>)` encoding will silently fail to
validate.

## Client configuration

### Cursor (`.cursor/mcp.json` or `~/.cursor/mcp.json`)

```json
{
  "mcpServers": {
    "wendRAG": {
      "type": "http",
      "url": "http://localhost:3000/mcp",
      "headers": {
        "Authorization": "Bearer wrag_your_key_here"
      }
    }
  }
}
```

Cursor (as of the version referenced in the user's `mcp.json` example) also
accepts an `api_key` shorthand:

```json
{
  "mcpServers": {
    "wendRAG": {
      "type": "url",
      "url": "http://localhost:3000/mcp",
      "api_key": "wrag_your_key_here"
    }
  }
}
```

### Claude Desktop

Claude Desktop supports MCP over stdio natively; for remote HTTP servers, use
an HTTP-capable client or wrap wendRAG behind a local stdio bridge.

### Raw HTTP (curl, integration tests)

```bash
curl -X POST http://localhost:3000/mcp \
  -H "Authorization: Bearer wrag_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"initialize","params":{},"id":1}'
```

See [`mcp-client-setup.md`](mcp-client-setup.md) for a complete walkthrough of
every major MCP client environment.

## Operational recommendations

- **Prefer distinct keys per client.** If key `ci-runner` is compromised, you
  can revoke it without taking every other client offline.
- **Rotate keys periodically.** Generate a replacement, configure clients with
  the new key, then revoke the old one.
- **Terminate TLS at a reverse proxy.** wendRAG speaks plain HTTP; put it
  behind nginx, Caddy, or a cloud load balancer to get TLS.
- **Restrict the listener.** Bind to `127.0.0.1` (`WEND_RAG_HOST=127.0.0.1`) if
  only local clients should connect.
- **Audit the keys file.** `keys.json` is the ground truth for who can talk to
  your server -- include it in your backup and access-review processes.

## Troubleshooting

### `401 Unauthorized` even though the key looks right

1. Confirm the header prefix is exactly `Bearer ` (one space).
2. Check for trailing whitespace or line breaks in the token.
3. Restart the daemon after editing `keys.json` or `WEND_RAG_API_KEY`; keys
   are loaded at startup.
4. Run `wend-rag key:list` to verify the key's prefix matches what the client
   is sending.

### Startup warns that authentication is disabled

This is expected when no keys are configured. Either:

- Run `wend-rag key:generate` to register a file-based key, or
- Set `WEND_RAG_API_KEY` before launching the daemon.

### `could not determine keys file location`

The daemon could not find a home/config directory. Set `WEND_RAG_KEYS_FILE`
to an explicit absolute path and retry.
